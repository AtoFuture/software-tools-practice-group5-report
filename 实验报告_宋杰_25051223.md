# 实验一 网络软件基础

## 1. 实验目的

本次实验旨在安装并运行 Cisco Packet Tracer 网络模拟软件，通过搭建简单的局域网拓扑，理解 C/S、B/S、P2P 等不同网络软件架构的区别，并掌握基本的网络配置和连通性测试方法。

## 2. 实验环境

- 操作系统：Windows 11
- 软件版本：Cisco Packet Tracer
- 网络拓扑：单交换机 2960 + 2 台 PC

## 3. 实验过程

### 步骤一：下载与安装 Cisco Packet Tracer

访问 Cisco NetAcad 官网，注册账号后下载 Windows 版本安装包

### 步骤二：搭建局域网拓扑

从左侧设备工具栏中拖拽 1 台交换机（2960 型号）和 2 台 PC（PC-PT）到中央工作区。点击设备工具栏中的 Connections，选择 Copper Straight-Through，依次将 PC1 和 PC2 连接到交换机的端口。

![](source/images/2026-07-11-15-29-36.png)

### 步骤三：配置 IP 地址

双击 PC1，在 Desktop 选项卡中点击 IP Configuration，设置 IP 为 192.168.1.1，子网掩码 255.255.255.0。PC2 设置为 192.168.1.2。

![](source/images/2026-07-11-15-30-28.png)

### 步骤四：测试网络连通性

在 PC1的Command Prompt中执行ping 192.168.1.2。

![](source/images/2026-07-11-15-31-37.png)

## 4. 实验结果

ping 连通性测试：从PC1(192.168.1.1)到PC2 (192.168.1.2)成功，收到Reply

![](source/images/2026-07-11-15-31-04.png)

## 5. 知识总结

- C/S 架构（客户端/服务器）：需安装专用客户端，客户端与服务器分工协作。典型例子：微信、网络游戏、Outlook。

- B/S 架构（浏览器/服务器）：只需浏览器即可访问，核心功能在服务器端。典型例子：淘宝、知乎、在线办公软件。

- P2P 架构（点对点）：每个节点既是客户端也是服务器，可直接通信。典型例子：BT下载、部分局域网通讯工具。

## 6. 出现问题与解决

- 一开始语言选项和指导手册里面的有出入————通过一个个找，尝试

## 7. 心得体会

这个实验主要是为我们展示了网络上面信息的传输过程，一些有用的指令比如ping，在后面学习的时候也有用到，总的来说还是在做基础。

# 实验二 Web 软件开发环境

## 1. 实验目的

本次实验旨在安装 Hexo 静态博客框架，了解和掌握 Web 开发环境的基础安装与配置（Node.js、npm、Git），理解 B/S 架构特点及静态站点与动态站点的区别。

- 操作系统：Windows 11
- 软件版本：Node.js，npm，Hexo CLI

从 Node.js 官网下载 LTS 版本 Windows 安装包，双击运行。在安装过程中务必勾选和"Add to PATH"选项。安装完成后重启命令提示符，执行验证命令。

```bash
node -v
npm -v
```

![](source/images/2026-07-11-15-42-17.png)

### 步骤二：配置 npm 镜像源

在命令提示符中执行以下命令，切换为国内镜像源以加速依赖下载。

### 步骤三：安装 Git for Windows

从 Git 官网下载安装包，安装完成后验证：

```bash
git --version
```

配置 Git 全局用户信息：

```bash
git config --global user.name "你的姓名"
git config --global user.email "你的邮箱@example.com"
```

![](source/images/2026-07-11-15-44-04.png)

### 步骤四：安装 Hexo CLI

在命令提示符中执行：

```bash
cd my-blog
npm install
```

## 5. 知识总结

- 静态博客：在本地将 Markdown 预编译为纯 HTML 页面，加载速度快、安全性高、无需数据库。典型代表：Hexo、Hugo、VuePress、Jekyll。

- 动态博客：每次用户访问时由服务器端程序动态生成页面，功能丰富、有后台管理界面、需要数据库。典型代表：WordPress、Typecho、Halo。

- Web 开发环境核心组件：运行时环境（Node.js）、包管理器（npm）、版本控制（Git）、构建工具（Hexo CLI）、本地服务器（Hexo server）。

- hexo server 启动后端口 4000 被占用：使用 hexo server -p 4001 解决。

本次实验旨在安装Visual Studio Code，安装和配置Markdown相关插件，学习Markdown基础语法，使用VS Code完成Markdown文档的编辑和渲染任务。

- Hexo 是一个基于 Node.js 的快速、简洁且高效的博客框架，使用 Markdown 解析文章，拥有 400+ 主题和丰富的插件生态。

- hexo init 速度极慢或失败————网络问题无法访问 GitHub，使用镜像源。

- git clone 主题失败————网络问题，改用 npm install hexo-theme-next 安装。

## 7. 心得体会

这个实验是 Web 网站的构建，这个对于以后学术交流和互通是比较有作用的，比如未来Github上面的下载和上传。此外，这个内容和实验三的内容是紧密关联的。这是我们 web 网页建立的基础内容。

# 实验五 虚拟机安装与使用

## 1. 实验目的

本次实验旨在使用虚拟机工具（VirtualBox）安装Linux系统（Ubuntu Server），通过克隆技术构建一个多机虚拟化环境，了解和掌握基于虚拟机的开发环境的基本构建方法和配置。

## 2. 实验环境

- 操作系统：Windows 11
- 虚拟机软件：Oracle VirtualBox
- 虚拟机系统：Ubuntu Server LTS
- 虚拟机配置：内存 2GB，硬盘 20GB
- 网络配置：双网卡（NAT + Host-Only）
- 网络拓扑：Host-Only 网络，IP 段 192.168.56.0/24
- 多机环境：template (192.168.56.101)，node1 (192.168.56.102)，node2 (192.168.56.103)，node3 (192.168.56.104)

## 3. 实验过程

### 步骤一：下载与安装 VirtualBox

访问 VirtualBox 官网（https://www.virtualbox.org/），下载 Windows 版本安装包并安装。安装时保持默认选项全选，安装完成后重启电脑。

![](source/images/2026-07-11-16-18-08.png)

### 步骤二：下载 Ubuntu Server ISO 镜像

从清华大学镜像站下载 ubuntu-22.04.5-live-server-amd64.iso 文件。

### 步骤三：创建模板虚拟机

打开 VirtualBox，点击"新建"按钮，按以下配置创建虚拟机：

- 名称：ubuntu-template
- ISO 映像：选择下载的Ubuntu Server ISO文件
- 类型：Linux，版本：Ubuntu (64-bit)
- 内存：2048 MB
- 硬盘：20 GB，VDI格式，动态分配

![](source/images/2026-07-11-16-18-51.png)

### 步骤四：配置虚拟机网络（双网卡）

在虚拟机设置中配置双网卡：

- 网卡 1（NAT）：用于虚拟机上网和下载软件

#### 步骤七：使用 rsync 同步文件

在 template 上执行以下命令练习 rsync 同步：

安装 rsync（如果未安装）：

```bash
sudo apt install rsync -y
```

创建测试目录和文件：

```bash
mkdir -p ~/dev/source
mkdir -p ~/dev/backup
echo "source file" > ~/dev/source/file1.txt
echo "source file2" > ~/dev/source/file2.txt
```

本地同步：

```bash
rsync -avz ~/dev/source/ ~/dev/backup/
ls ~/dev/backup/
```

推送到远程（template → node1）：

```bash
rsync -avz -e ssh ~/dev/source/ ryu@node1:~/dev/source/
ssh ryu@node1 "ls -l ~/dev/source/"
```

从远程拉取（node1 → template）：

```bash
rsync -avz -e ssh ryu@node1:~/dev/source/ ~/dev/source_copy/
ls ~/dev/source_copy/
```

保持完全一致（--delete 会删除目标中多余的文件）：

先在 node1 目标目录创建一个多余文件：

```bash
ssh ryu@node1 "echo 'extra file' > ~/dev/source/extra.txt"
ssh ryu@node1 "ls -l ~/dev/source/"
```

执行带 --delete 的同步：

```bash
rsync -avz --delete -e ssh ~/dev/source/ ryu@node1:~/dev/source/
ssh ryu@node1 "ls -l ~/dev/source/"
```

排除特定文件（--exclude）：

```bash
mkdir -p ~/dev/logs
echo "log content" > ~/dev/logs/app.log
echo "data content" > ~/dev/logs/data.txt

rsync -avz --exclude='*.log' -e ssh ~/dev/ ryu@node2:~/dev/
ssh ryu@node2 "ls -l ~/dev/logs/"
```

## 5. 知识总结

- SSH（Secure Shell）：一种加密网络协议，用于在不安全的网络中安全地远程登录和执行命令，默认使用 22 端口。

- SSH 两种认证方式：
  - 密码认证：每次登录输入密码，安全性较低，适用于临时访问
  - 密钥认证：使用公钥/私钥对验证身份，安全性高，适用于日常管理、自动化脚本

- SCP：基于 SSH 的简单文件复制工具，适用于一次性、少量文件传输

- rsync：增量同步工具，只传输变化的部分，适用于目录同步、定期备份

- SSH 密钥配置流程：生成密钥对（ssh-keygen）→ 复制公钥到目标服务器（ssh-copy-id）→ 测试免密登录（ssh user@host）

## 6. 出现问题与解决

- 克隆后的虚拟机（node1/node2）网络不通，SSH 连接报 No route to host————node1/node2未开机

- SCP 上传文件到家目录报 Permission denied————不传到家目录，改用 /tmp 作为中转

- Permission denied (publickey,password)：密码错误或密钥未正确配置，检查用户名密码，检查 authorized_keys 配置

- rsync 的 --delete 测试报错 No such file or directory————目录不存在，先创建目录再写文件

#### 步骤一：准备工作确认

在 Windows 命令提示符中进入 Hexo 博客项目目录：

cd C:\Users\abc\Desktop\school\短学期\my blog\my-blog

确认 Hexo 博客正常运行：

```bash
hexo server
```

确认 Git 配置：

```bash
git config --global user.name
git config --global user.email
```

如果未配置，执行：

```bash
git config --global user.name "你的GitHub用户名"
git config --global user.email "你的邮箱@example.com"
```

#### 步骤二：创建两个 GitHub 仓库

登录 GitHub 账号，创建两个仓库：

源码仓库（my-blog）：用于存放 Hexo 博客源码，可见性选择 Private（私有）

部署仓库（用户名.github.io）：用于托管静态页面，可见性选择 Public（公开）

注意：部署仓库名称必须严格按照 你的GitHub用户名.github.io 格式命名。

#### 步骤三：将本地项目关联到源码仓库

在 Windows 命令提示符中（Hexo 项目目录下）执行：

```bash
git init
git remote add origin https://github.com/你的用户名/my-blog.git
```

创建 .gitignore 文件：

```bash
echo node_modules/ > .gitignore
echo .DS_Store >> .gitignore
echo public/ >> .gitignore
echo .deploy_git/ >> .gitignore
```

提交并推送：

```bash
git add .
git commit -m "初始化Hexo博客源码"
git branch -M main
git push -u origin main
```

推送时提示输入密码，使用 Personal Access Token（稍后生成）作为密码。

![](source/images/2026-07-11-17-18-27.png)

#### 步骤四：配置 GitHub Actions

在 Hexo 项目目录下创建工作流目录和文件：

```bash
mkdir .github
mkdir .github\workflows
```

创建 .github\workflows\deploy.yml 文件，内容如下（请修改 external_repository 中的用户名）：

```yaml
name: Deploy to GitHub Pages

on:
  push:
    branches: [main]
  workflow_dispatch:

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - name: 检出源码
        uses: actions/checkout@v3
      - name: 安装Node.js
        uses: actions/setup-node@v3
        with:
          node-version: '20'
      - name: 缓存依赖
        uses: actions/cache@v3
        with:
          path: ~/.npm
          key: ${{ runner.os }}-node-${{ hashFiles('**/package-lock.json') }}
      - name: 安装依赖
        run: npm install
      - name: 生成静态文件
        run: npx hexo generate
      - name: 创建.nojekyll文件
        run: echo "" > public/.nojekyll
      - name: 部署到GitHub Pages
        uses: peaceiris/actions-gh-pages@v3
        with:
          personal_token: ${{ secrets.PERSONAL_TOKEN }}
          publish_dir: ./public
          publish_branch: main
          external_repository: 你的用户名/你的用户名.github.io
```

提交并推送：

```bash
git add .github/workflows/deploy.yml
git commit -m "添加GitHub Actions自动部署"
git push
```

#### 步骤五：生成并配置 Personal Access Token

在 GitHub 网页中操作：

点击右上角头像 -> Settings -> Developer settings -> Personal access tokens -> Tokens (classic)

点击 Generate new token -> Generate new token (classic)

填写信息：
- Note：Hexo Deploy
- Expiration：No expiration
- 勾选 repo 权限

点击 Generate token，立即复制生成的 Token（以 ghp_ 开头）

在源码仓库（my-blog）中：Settings -> Secrets and variables -> Actions -> New repository secret
- Name：PERSONAL_TOKEN
- Secret：粘贴刚才复制的 Token

#### 步骤六：配置 GitHub Pages

在部署仓库（用户名.github.io）中：Settings -> Pages -> Branch: main -> Save

等待几分钟后访问：https://你的用户名.github.io

### 第二部分：部署到虚拟机（node1）

#### 步骤七：安装和配置 Nginx

通过 SSH 连接到 node1：

```bash
ssh ryu@192.168.56.102
```

更新软件包并安装 Nginx：

```bash
sudo apt update
sudo apt install nginx -y
sudo systemctl start nginx
sudo systemctl enable nginx
```

创建站点目录：

```bash
sudo mkdir -p /var/www/blog
sudo chown -R ryu:ryu /var/www/blog
```

创建 Nginx 配置文件：

```bash
sudo nano /etc/nginx/sites-available/blog
```

写入以下内容：

```nginx
server {
    listen 80;
    server_name _;
    root /var/www/blog;
    index index.html;
}
```

启用站点并禁用默认站点：

```bash
sudo ln -s /etc/nginx/sites-available/blog /etc/nginx/sites-enabled/
sudo rm /etc/nginx/sites-enabled/default
sudo nginx -t
sudo systemctl reload nginx
```

## 5. 知识总结

- 持续集成（CI）：代码变更后自动进行构建和测试。本实验中，推送代码后自动执行 hexo generate。

- 持续部署（CD）：构建通过后自动部署到生产环境。本实验中，静态文件自动推送到 GitHub Pages。

- 本实验整体架构：开发环境（Windows + Hexo）→ 版本控制（Git + GitHub）→ CI/CD（GitHub Actions）→ 生产环境（GitHub Pages + Nginx）

- 部署方式对比：GitHub Pages 提供公网访问，适合个人博客展示；Nginx 部署在虚拟机，适合内网环境或自定义服务器配置。

## 6. 出现问题与解决

- GitHub 打不开————DNS 污染，要自己搜ipv4，通过 host 映射

- 虚拟机 Nginx 显示默认欢迎页，而不是博客内容————禁用默认站点并启用博客站点

- SCP 上传 public/ 目录到虚拟机时报权限错误————在 node1 上修改目录所有者
