实验目标

查看已有账户，创建实验用户和用户组，配置 Home、共享目录、测试文件与 SSH 关键文件权限，并验证组权限。

关键操作与最终配置

id hdu

创建 webteam 组和 webstudent 账户；账户 Home 为 /home/webstudent，Shell 为 /bin/bash，并将 hdu 与 webstudent 加入 webteam。

问题分析

最初 hdu 访问 /home/webstudent/course-share 时出现 Permission denied。原因是上级目录所属组和当前 SSH 会话的组状态未正确刷新。将 /home/webstudent 所属组改为 webteam、权限设为 750，并退出 SSH 重新登录后，hdu 可成功访问共享目录。

六、实验六 Linux 环境配置

实验目标

分别配置系统级和用户级 Shell 环境变量，理解配置文件作用范围，并通过重新登录验证持久化。

系统级与用户级配置

/etc/profile.d/software-tools-practice.sh 的所有者和权限为 root:root、644：

export WEB_ROOT="/var/www/html"
export REPORT_PATH="/var/www/html/report"
export COURSE_ENV="software-tools-practice"

在当前用户的 ~/.bashrc 中加入：

export USER_WORKSPACE="$HOME/software-tools-practice"
alias cpractice='cd "$USER_WORKSPACE"'

/etc/profile.d/ 适合登录 Shell 的系统级统一变量；~/.bashrc 只影响当前用户的 Bash 交互环境；Ubuntu 的 ~/.profile 会按条件加载 ~/.bashrc。使用 source 只证明当前会话可加载，因此还要重新登录验证自动加载与持久化。

WEB_ROOT=/var/www/html
REPORT_PATH=/var/www/html/report
COURSE_ENV=software-tools-practice
USER_WORKSPACE=/home/hdu/software-tools-practice

SSH 服务与端口转发

OpenSSH Server 已安装，服务状态为 enabled、active，Ubuntu 内部监听 22 端口。VirtualBox 将 Windows 127.0.0.1:2222 映射到 Ubuntu 22。

SSH 安全配置

port 22
pubkeyauthentication yes
passwordauthentication yes
maxauthtries 6

在 /etc/ssh/sshd_config.d/99-course-security.conf 中配置 PermitRootLogin no，再执行：

sudo sshd -t
sudo systemctl restart ssh
sudo systemctl is-active ssh

Ubuntu 中读取到 Remote management transfer test。

八、实验八 Web 软件部署

实验目标

使用实际存在且已成功运行的 deploy.ps1 完成 Hexo 构建、SSH/SCP 上传、Nginx 部署、权限设置、配置检查和 HTTP 验证。

Nginx 与部署目录

站点目录为 /var/www/html/，报告入口为 /var/www/html/report/index.html。

自动部署脚本

脚本依次检查本地 Hexo，执行 clean 和 generate，检查生成入口，清理远程暂存目录，SCP 上传 public，复制到 Nginx 目录，为目录设 755、文件设 644，执行 nginx -t，reload Nginx，检查报告文件并验证 HTTP。

& $HexoCommand clean
& $HexoCommand generate
if (-not (Test-Path $ReportIndex)) { throw "Generated report was not found" }
& scp -P $SshPort -r $PublicDirectory $RemoteTarget
& ssh -tt -p $SshPort "${RemoteUser}@${ServerHost}" $RemoteCommand
& curl.exe -I "http://127.0.0.1:8080/report/"

HTTP 访问验证

Nginx deployment completed.
HTTP/1.1 200 OK
Deployment succeeded.

局域网发布与多终端访问

原规则 127.0.0.1:8080 仅供本机访问。新增 VirtualBox NAT 转发 HTTP-LAN,tcp,0.0.0.0,8081,,80，并新增 Windows 防火墙规则：

DisplayName: VirtualBox Nginx LAN 8081
Direction: Inbound
Action: Allow
Profile: Any

实验时 Windows 校园网 IPv4 为 10.150.103.10，局域网地址为 http://10.150.103.10:8081/report/。0.0.0.0:8081 表示 Windows 所有网络接口均可接收请求，再由 VirtualBox 转发到 Ubuntu 80 端口。Nginx 可处理多个终端请求，同一局域网设备可使用同一地址访问。

当前已经完成局域网地址访问验证，并具备多终端同时访问条件；现有证据未显示三台设备同时出现在同一画面，因此不作“三台设备已同时访问”的表述。本节是局域网发布，不是公网发布。

问题分析与结果验证

首次部署后曾出现样式未完整加载的问题。重新执行 Hexo 清理与生成，并上传完整 public 目录后恢复。中文乱码通过 UTF-8 保存及 <meta charset="UTF-8" /> 解决。最终本机和校园网局域网地址均完成访问验证。

十、技术展望

主题：将 Hexo 手工部署升级为 GitHub Actions + SSH 自动化部署

当前方案是在 Windows 本地维护 Hexo 项目，手动运行 deploy.ps1，通过 SSH/SCP 上传到 Ubuntu，再由 Nginx 发布。该方案可用，但需要人工启动脚本和输入认证信息，依赖本地环境，也可能因漏执行生成或上传步骤造成版本不一致。

可调研 GitHub Actions、GitLab CI/CD、Jenkins、Gitee Go，并比较 VitePress、Hugo、Caddy。结合现有 GitHub、Hexo、SSH 和 Nginx 环境，最终升级方向选择 GitHub Actions + SSH Key + Nginx。

升级流程

  1. 向 main 分支推送经检查的源码。
  2. Actions 检出代码并安装指定 Node.js。
  3. 执行 npm ci 和 Hexo 生成。
  4. 将部署专用 SSH 私钥存入 GitHub Secrets，不写入仓库。
  5. 使用 rsync 或 scp 上传到带时间戳的暂存目录。
  6. 执行 nginx -t，通过后切换版本并 reload。
  7. 执行 HTTP 检查；任一步失败立即停止。

风险、回滚与验收

风险包括私钥泄露、Secrets 配置错误、上传中断、Nginx 配置错误和 Actions 权限不足。应使用最小权限部署账户和密钥，限制部署分支，不在日志中输出敏感值，并保留部署历史。

远程目录按时间戳保存 release，以软链接指向当前版本。失败时恢复 previous，不直接删除旧版本。验收标准是：推送后自动运行、依赖安装和构建成功、页面正常、HTTP 200、不需要手工 SCP。

调研方法与参考资料

调研方法包括阅读官方文档、检索 Actions Marketplace、分析开源 workflow、查阅技术社区文章和对比实验。

  1. GitHub Actions：Deployments and environments
  2. GitHub Actions：Using secrets in GitHub Actions
  3. Hexo 官方文档
  4. Hexo 命令文档
  5. Nginx 官方文档
  6. OpenSSH sshd_config 手册
  7. Git 官方参考文档

十一、实践总结、心得体会与意见建议

客观实践总结

本次实践形成了可验证流程：在 Windows 配置开发工具，使用 VS Code 编写报告并由 Git 管理；在 VirtualBox 安装 Ubuntu Server，完成 Linux 基本操作、账户权限和环境配置；通过 SSH 与 SCP 远程管理；由 Hexo 生成静态文件并使用 Nginx 发布；最后通过本机和校园网局域网地址验证访问。Packet Tracer 作为八个正式实验之外的扩展实践单独记录。

实验心得体会

这次实践刚开始的时候，我以为主要就是安装软件、照着手册输命令，再把截图放进报告里。真正做下来以后才发现，八个实验其实是连在一起的。前面搭建 Windows 开发环境和 Hexo 项目，后面又要把网页传到 Ubuntu，再通过 Nginx 发布，少了任何一步，最后的网站都跑不起来。

整个过程中我遇到的问题比预想中多。最明显的一次是我启动虚拟机之后，没有先通过 SSH 进入 Ubuntu，就直接在 Windows PowerShell 里执行了 uname、grep、apt 等 Linux 命令，结果全部报错。当时我还以为是虚拟机或者命令有问题，后来才发现自己根本没有进入 Linux 环境。配置 Linux 账户权限时，我给共享目录设置了用户组和权限，但 hdu 访问时还是出现 Permission denied，最后检查才发现上级 Home 目录的所属组和当前 SSH 会话的用户组状态都会影响访问，修改权限并重新登录后才正常。后面写 deploy.ps1 时又遇到了中文乱码和 here-string 被提前截断的问题，脚本看起来写好了，实际上有一部分命令却直接在 PowerShell 中运行了。这个过程让我意识到，看到报错不能只反复重试，应该先确认自己当前在哪个环境，再检查路径、权限、端口、编码和服务状态。

最后部署脚本能够自动执行 Hexo 构建、SCP 上传、Nginx 配置检查和 HTTP 状态验证，并成功返回 HTTP/1.1 200 OK。我还通过校园网地址打开了部署在 Ubuntu 中的网页。相比一开始只会在本地打开 HTML，现在我对从代码编辑、版本管理、文件传输到服务器部署的整个流程有了更完整的理解。

这次实践给我最大的感受是，很多工具单独看并不难，真正麻烦的是把它们连接起来，并且保证每一个环节都正确。Git、VirtualBox、Linux、SSH 和 Nginx 以前在我看来是几块分开的内容，这次做完后才真正理解它们在一个项目里分别负责什么。以后再遇到类似问题，我应该会比以前更愿意根据报错一步一步定位，而不是直接重装或者从头再做。

意见与建议

我觉得这门实践课的内容比较实用，特别是 Linux 权限、SSH 远程登录和 Web 部署这些部分，和以后真正做项目有比较直接的联系。以前学编程时更多关注代码本身，这次能够看到网页怎样从本地文件变成服务器上可以访问的站点，体验还是比较完整的。

不过八个实验集中在一周左右完成，后半部分内容比较密集，Linux 账户权限、环境变量、端口转发和部署很容易混在一起。建议以后可以提前提供一个统一的目录结构和最终成果示例，并在课堂上完整演示一次“本地生成网页—SSH 登录—SCP 上传—Nginx 部署—其他设备访问”的全过程。GitHub 小组协作方面也可以增加一次分支创建、Pull Request、冲突处理和合并的演示，这样组员各自在不同分支提交材料时会更清楚，也不容易把文件传到错误目录。
