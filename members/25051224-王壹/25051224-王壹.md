## 五、实验五 Linux 账户与权限管理

### 5.1 实验目标

查看已有账户信息，创建实验用户和用户组，配置 Home 目录、共享目录、测试文件与 SSH 关键文件权限，并验证组权限生效。

### 5.2 账户、组和目录配置

首先查看已有 `hdu` 用户：

```bash
id hdu
getent passwd hdu
echo $HOME
groups hdu
ls -ld /home/hdu
```

随后创建 `webteam` 组和 `webstudent` 账户。`webstudent` 的 Home 为 `/home/webstudent`，Shell 为 `/bin/bash`；将 `hdu` 和 `webstudent` 加入 `webteam`。

实验目录和文件的最终配置为：

| 对象 | 所有者与组 | 权限 |
|---|---|---:|
| `/home/webstudent` | `webstudent:webteam` | `750` |

| `account-test.txt` | `webstudent:webteam` | `640` |
| `~/.ssh` | hdu 用户 | `700` |
| `~/.ssh/authorized_keys` | hdu 用户 | `600` |

![Linux 用户与目录权限初步验证](./实验截图/19-linux-user-and-permission.png)

*图 19 Linux 用户与目录权限初步验证。截图显示普通用户、sudo 和目录权限信息，证明账户与 Web 目录权限可以被查询。*

### 5.3 问题分析

最初 `hdu` 访问 `/home/webstudent/course-share` 时出现 `Permission denied`。虽然 `course-share` 属于 `webteam`，但上级目录 `/home/webstudent` 的所属组和当前 SSH 会话中的用户组状态未正确刷新，导致 `hdu` 无法穿越上级目录。

处理步骤为：将 `/home/webstudent` 的所属组改为 `webteam`，权限设为 `750`；退出 SSH 后重新登录，使新增组身份生效。重新验证后，`hdu` 可以访问共享目录。

### 5.4 结果验证

账户、组、Home、共享目录、测试文件和 SSH 关键文件权限均达到预期，权限错误的原因和处理过程有客观记录。

## 六、实验六 Linux 环境配置

### 6.1 实验目标

分别配置系统级和用户级 Shell 环境变量，理解 `/etc/profile.d/`、`~/.profile` 与 `~/.bashrc` 的作用范围，并通过重新登录验证配置持久化。

### 6.2 系统级配置

创建 `/etc/profile.d/software-tools-practice.sh`，所有者和权限为 `root:root`、`644`，内容为：

```bash
export WEB_ROOT="/var/www/html"
export REPORT_PATH="/var/www/html/report"
export COURSE_ENV="software-tools-practice"
```

`/etc/profile.d/` 中的脚本面向登录 Shell 的系统级环境初始化，适合配置需要由多个用户读取的统一变量。

### 6.3 用户级配置

在 `~/.bashrc` 中加入：

```bash
export USER_WORKSPACE="$HOME/software-tools-practice"
alias cpractice='cd "$USER_WORKSPACE"'
```

`~/.bashrc` 只影响当前用户的 Bash 交互环境；Ubuntu 默认的 `~/.profile` 会在登录时按条件加载 `~/.bashrc`。使用 `source` 可在当前会话立即加载修改，但仅执行 `source` 不能证明新登录会话也会自动加载，因此还需要退出 SSH 并重新登录。

验证结果为：

```text
WEB_ROOT=/var/www/html
REPORT_PATH=/var/www/html/report
COURSE_ENV=software-tools-practice
USER_WORKSPACE=/home/hdu/software-tools-practice
```

### 8.3 自动部署脚本

`deploy.ps1` 的流程为：检查项目本地 Hexo，执行 `clean` 和 `generate`，检查 `public/report/index.html`，清理远程临时目录，SCP 上传 `public`，复制到 `/var/www/html`，为目录设置 `755`、为文件设置 `644`，执行 `nginx -t`，重新加载 Nginx，检查报告文件并用 HTTP 请求验证状态。

报告仅摘录脚本关键逻辑：

```powershell
& $HexoCommand clean
& $HexoCommand generate
if (-not (Test-Path $ReportIndex)) { throw "Generated report was not found" }
& scp -P $SshPort -r $PublicDirectory $RemoteTarget
& ssh -tt -p $SshPort "${RemoteUser}@${ServerHost}" $RemoteCommand
& curl.exe -I "http://127.0.0.1:8080/report/"
```

原规则 `127.0.0.1:8080` 仅供 Windows 本机访问。为在校园网局域网内验证访问，新增 VirtualBox NAT 转发：

```text
HTTP-LAN,tcp,0.0.0.0,8081,,80
```

同时新增 Windows 防火墙规则：

```text
DisplayName: VirtualBox Nginx LAN 8081
Direction: Inbound
Action: Allow
Profile: Any
```

实验时 Windows 校园网 IPv4 地址为 `10.150.103.10`，局域网访问地址为 `http://10.150.103.10:8081/report/`。`0.0.0.0:8081` 表示 Windows 所有网络接口均可接收 8081 端口请求，再由 VirtualBox 转发到 Ubuntu 的 80 端口。Nginx 能同时处理多个终端请求，因此同一局域网中的设备可使用同一地址访问，不需要为每台设备单独设置端口。

## 十、技术展望

### 10.1 主题：将 Hexo 手工部署升级为 GitHub Actions + SSH 自动化部署

当前方案是在 Windows 本地维护 Hexo 项目，手动运行 `deploy.ps1`，通过 SSH/SCP 上传到 Ubuntu，再由 Nginx 发布。该方案已经能够完成部署，但仍需要人工启动脚本和输入认证信息，依赖本地电脑环境，也可能因漏执行生成或上传步骤造成版本不一致。

可调研的替代方案包括 GitHub Actions、GitLab CI/CD、Jenkins 和 Gitee Go；站点工具还可比较 VitePress、Hugo，Web 服务可继续使用 Nginx或调研 Caddy 自动 HTTPS。结合现有 GitHub 仓库、Hexo 源码、SSH 和 Nginx 环境，最终升级方向选择 **GitHub Actions + SSH Key + Nginx**。

### 10.2 升级流程

1. 向 `main` 分支推送经检查的源码。
2. GitHub Actions 检出代码并安装指定版本的 Node.js。
3. 执行 `npm ci`，按锁文件安装依赖。
4. 执行 `npx hexo clean` 和 `npx hexo generate`。
5. 将部署专用 SSH 私钥存入 GitHub Actions Secrets，不写入仓库。
6. 使用 `rsync` 或 `scp` 上传到带时间戳的远程暂存目录。
7. 服务器执行 `nginx -t`，检查通过后切换当前版本并 reload Nginx。
8. 对报告地址执行 HTTP 检查；若构建、上传或配置检查失败，立即中止部署。

### 10.3 风险、回滚与验收

主要风险包括私钥泄露、Secrets 配置错误、上传中断、Nginx 配置错误和 Actions 权限不足。应使用权限最小化的部署账户和密钥，限制可部署分支，不在日志中输出敏感值，并保留部署历史。

远程目录可按时间戳保存 release，以软链接指向当前版本。部署失败时恢复 `previous` 指向的上一版本，不直接删除旧版本。验收标准为：推送后工作流自动运行、依赖安装和 Hexo 构建成功、页面正常显示、HTTP 返回 200、整个流程不需要手工 SCP。

### 10.4 调研方法与参考资料

调研方法包括阅读官方文档、检索 GitHub Actions Marketplace、分析开源项目 workflow、查阅技术社区文章并进行对比实验。参考资料采用以下可核验来源：

## 十一、实践总结、心得体会与意见建议

### 11.1 客观实践总结

本次实践形成了以下可验证流程：在 Windows 配置 Web 开发工具，使用 VS Code 编写报告并由 Git 管理版本；在 VirtualBox 中安装 Ubuntu Server，完成 Linux 基本操作、账户权限和环境变量配置；通过 SSH 与 SCP 远程管理；由 Hexo 生成静态文件并使用 Nginx 发布；最后分别通过本机地址和校园网局域网地址验证访问。Packet Tracer 内容作为正式八个实验之外的扩展实践单独记录。

### 11.2 实验心得体会
这次实践刚开始的时候，我以为主要就是安装软件、照着手册输命令，再把截图放进报告里。真正做下来以后才发现，八个实验其实是连在一起的。前面搭建 Windows 开发环境和 Hexo 项目，后面又要把网页传到 Ubuntu，再通过 Nginx 发布，少了任何一步，最后的网站都跑不起来。

整个过程中我遇到的问题比预想中多。最明显的一次是我启动虚拟机之后，没有先通过 SSH 进入 Ubuntu，就直接在 Windows PowerShell 里执行了 uname、grep、apt 等 Linux 命令，结果全部报错。当时我还以为是虚拟机或者命令有问题，后来才发现自己根本没有进入 Linux 环境。配置 Linux 账户权限时，我给共享目录设置了用户组和权限，但 hdu 访问时还是出现 Permission denied，最后检查才发现上级 Home 目录的所属组和当前 SSH 会话的用户组状态都会影响访问，修改权限并重新登录后才正常。后面写 deploy.ps1 时又遇到了中文乱码和 here-string 被提前截断的问题，脚本看起来写好了，实际上有一部分命令却直接在 PowerShell 中运行了。这个过程让我意识到，看到报错不能只反复重试，应该先确认自己当前在哪个环境，再检查路径、权限、端口、编码和服务状态。

最后部署脚本能够自动执行 Hexo 构建、SCP 上传、Nginx 配置检查和 HTTP 状态验证，并成功返回 HTTP/1.1 200 OK。我还通过校园网地址打开了部署在 Ubuntu 中的网页。相比一开始只会在本地打开 HTML，现在我对从代码编辑、版本管理、文件传输到服务器部署的整个流程有了更完整的理解。

这次实践给我最大的感受是，很多工具单独看并不难，真正麻烦的是把它们连接起来，并且保证每一个环节都正确。Git、VirtualBox、Linux、SSH 和 Nginx 以前在我看来是几块分开的内容，这次做完后才真正理解它们在一个项目里分别负责什么。以后再遇到类似问题，我应该会比以前更愿意根据报错一步一步定位，而不是直接重装或者从头再做。


### 11.3 意见与建议

我觉得这门实践课的内容比较实用，特别是 Linux 权限、SSH 远程登录和 Web 部署这些部分，和以后真正做项目有比较直接的联系。以前学编程时更多关注代码本身，这次能够看到网页怎样从本地文件变成服务器上可以访问的站点，体验还是比较完整的。

不过八个实验集中在一周左右完成，后半部分内容比较密集，Linux 账户权限、环境变量、端口转发和部署很容易混在一起。建议以后可以提前提供一个统一的目录结构和最终成果示例，并在课堂上完整演示一次“本地生成网页—SSH 登录—SCP 上传—Nginx 部署—其他设备访问”的全过程。GitHub 小组协作方面也可以增加一次分支创建、Pull Request、冲突处理和合并的演示，这样组员各自在不同分支提交材料时会更清楚，也不容易把文件传到错误目录。
